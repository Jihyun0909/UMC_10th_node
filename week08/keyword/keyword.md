## 1. Swagger & OpenAPI

### 정의
* **OpenAPI**: RESTful API의 엔드포인트, 데이터 구조, 응답 포맷 등을 기술하기 위해 규정된 기계 가독형 인터페이스 기술 규격(Specification)이다.
* **Swagger**: OpenAPI 사양을 기반으로 API를 설계, 빌드, 문서화 및 소비할 수 있도록 지원하는 실질적인 오픈소스 도구 체인(Swagger UI, Swagger Editor 등)의 집합이다.

### 주요 속성
* **언어 독립성(Language-Agnostic)**: 특정 프로그래밍 언어가 아닌 YAML 또는 JSON 포맷을 기반으로 하므로, 백엔드 기술 스택과 무관하게 공통 언어로 명세서를 정의할 수 있다.
* **대화형 인터페이스(Interactive UI)**: 웹 브라우저 환경에서 포스트맨 같은 별도의 외부 도구 없이 `Try it out` 버튼을 통해 API를 즉시 테스트할 수 있는 샌드박스를 제공한다.

### 장단점
* **장점**: 프론트엔드와 백엔드 간의 정적 명세 공유를 통해 커뮤니케이션 비용을 효율적으로 단축한다.
* **단점**: 소스 코드 수정 시 문서(YAML/JSON 주석 등)를 수동으로 갱신하지 않으면, 구현체와 명세 문서 간의 정합성이 무너지는 동기화 문제가 발생한다.

## 2. TSOA (TypeScript-first OpenAPI)

### 정의
* TypeScript의 타입 정의와 데코레이터(Decorator) 문법을 컴파일 기준으로 삼아, OpenAPI 사양 문서 및 Express 등 프레임워크의 라우팅 소스 코드를 자동으로 빌드 및 파생해 주는 프레임워크이다.



### 주요 속성
* **단일 진실 공급원(Single Source of Truth)**: 별도의 문서 파일 없이 오직 TypeScript 소스 코드 하나만을 유일한 인터페이스 명세 기준으로 삼는다.
* **컴파일 타임 생성(Static Generation)**: 런타임 이전 빌드 프로세스 내에서 라우팅 장부인 `routes.ts`와 명세서 파일인 `swagger.json`을 동적으로 출력한다.

### 장단점
* **장점**: 코드 변경 사항이 문서에 즉각 반영되므로 Swagger 특유의 동기화 불일치 리스크를 완전히 제거한다. 입력 DTO에 대한 런타임 유효성 검증(Validation) 미들웨어가 자동으로 주입되어 안정성이 향상된다.
* **단점**: TSOA 고유의 데코레이터 문법에 종속되는 프레임워크 락인(Lock-in)이 발생하며, ECMAScript 모듈(ESM) 및 컴파일러 옵션 설정에 따라 초기 환경 구성 시 조율 비용이 발생한다.

## 3. Type-Driven-Documentation (타입 주도 문서화)

### 정의
* 자연어나 별도의 마크다운 문서 대신, 정적 컴파일 언어가 제공하는 고유의 '타입 시스템(Type System)'을 설계 중심에 두고 이를 가공 및 추출하여 소프트웨어 결과 문서로 자동 변환하는 개발 패러다임이다.

### 주요 속성
* **컴파일러 검증성(Compiler-verified)**: 문서의 모든 명세 구조가 컴파일러의 정적 분석 조건을 통과한 실질적 타입을 기반으로 하므로 휴먼 에러가 원천 차단된다.
* **코드-문서 일체화(Code as Documentation)**: 소스 코드 내부의 인터페이스 선언문 자체가 데이터 규격과 비즈니스 계약 조건을 나타내는 명세 역할을 수행한다.

### 장단점
* **장점**: 소스 코드 리팩토링으로 인한 데이터 형식 변경 시, 컴파일러가 의존성 손상 부위를 탐지하므로 명세 누락 및 유지보수 불일치 현상을 방지한다.
* **단점**: 초기 설계 단계에서 정교한 도메인 모델링과 엄격한 인터페이스 분리가 전제되어야 하므로 프로젝트 초기 진입 장벽(Learning Curve)과 아키텍처 설계 난이도가 상승한다.
