- ES
    
    자바스크립트 언어의 표준 규격이다. 브라우저마다 자바스크립트가 다르게 동작하면 안 되니까 “이렇게 만들자!”라고 약속한 가이드라인이다.
    
- ES6
    
    자바스크립트 역사상 가장 큰 변화가 일어난 버전이다.
    
    - ES6의 주요 변화 및 특징
        
        let, const 도입, 화살표 함수(⇒), 템플릿 리터럴(백틱 ` 사용), 구조 분해 할당, Spread 연산자(…), Promise 도입
        
    - ES6를 중요시 하는 이유
        
        가독성이 비약적으로 좋아졌으며, 현대적인 프레임워크(React, Vue, Node.js)들이 모두 ES6+ 문법을 기반으로 돌아가기 때문이다.
        
- ES Module
    
    import와 export를 사용하는 자바스크립트 공식 모듈 시스템이다. 예전에는 require를 썼지만, 이제는 브라우저와 서버(Node.js) 모두 이 방식을 표준으로 밀고 있다.
    
- TypeScript
    
    자바스크립트에 타입을 입힌 언어이다.
    
    장점: 코드를 실행 하기 전에 에러를 잡아준다, 대규모 프로젝트에서 협업할 때 남이 짠 코드가 어떤 데이터를 주고받는지 타입만 보고도 알 수 있어 유지보수에 용이하다.
    
- 프로젝트 아키텍처
    - 프로젝트 아키텍처가 중요한 이유
        
        집을 지을 때 설계도가 없으면 나중에 방을 하나 늘리려다 집이 무너질 수 있는 것처럼 소프트웨어에서 유지보수, 확장성, 테스트를 쉽게 하기 위해 미리 구조를 잘 짜두어야 한다.
        
    - Modular Monolith Architecture
        
        하나의 큰 서비스 안에 기능별로 모듈을 나눠놓은 구조이다. 관리하기 쉽고 초기에 빠르게 개발할 수 있다.
        
    - Service-Oriented Architecture(Service Layer Pattern)
        
        서비스들을 독립적인 단위로 나누어 서로 통신하게 하는 구조이다.
        
    - MVC 패턴
        
        Model(데이터), View(화면), Controller(로직 연결)로 역할을 나눈 고전적이고 강력한 패턴이다. 백엔드에서는 주로 View를 제외한 MC 구조를 많이 사용한다.
        
    - 그 외 다른 프로젝트 구조
        - 계층화 아키텍처 (Layered Architecture)
            - 가장 보편적인 구조로, 시스템을 유사한 책임을 가진 레이어로 분리한다.
            - **구성:** Presentation(Controller) ↔ Business(Service) ↔ Persistence(Repository) ↔ Database.
            - **특징:** 각 계층은 바로 아래 계층에만 의존한다. 구조가 단순하여 파악이 빠르지만, 규모가 커질수록 서비스 계층이 비대해지는 'Fat Service' 문제가 발생할 수 있다.
        - 마이크로서비스 아키텍처 (Microservices Architecture, MSA)
            - 하나의 거대한 애플리케이션(Monolith)을 기능 단위로 쪼개어 여러 개의 작은 서비스로 운영하는 방식이다.
            - **특징:** 각 서비스는 독립적인 DB와 서버를 가진다. 서비스 간 통신은 주로 REST API나 메시지 큐(RabbitMQ, Kafka)를 사용한다.
            - **장점:** 특정 기능만 독립적으로 배포하거나 확장하기 유리하지만, 서비스 간 통신 비용 및 관리 복잡도가 매우 높다.
        - 클린 아키텍처 (Clean Architecture)
            - 로버트 C. 마틴이 제안한 구조로, **비즈니스 로직이 외부 라이브러리나 프레임워크에 의존하지 않게** 만드는 것이 핵심이다.
            - **특징:** 의존성 방향이 항상 내부(Entity, Use Case)를 향한다. DB나 웹 프레임워크가 바뀌어도 핵심 로직은 수정할 필요가 없으므로 유지보수성이 극대화된다.
            - **단점:** 코드량이 많아지고 초기 설계 난이도가 매우 높습니다.
        
- 비즈니스 로직
    
    사용자의 요구사항을 해결하기 위한 실제 핵심 코드이다. 예를 들어, “사용자가 돈까스 미션을 완료하면 포인트 500점을 주고, 등급을 업데이트한다.” ← 이 규칙 자체가 비즈니스 로직이다.
    
- DTO
    
    계층 간 데이터를  주고받기 위해 사용하는 바구니 같은 객체이다. DB에서 가져온 민감한 정보(비밀번호 등)를 다 보여주지 않고, 화면에 딱 필요한 정보만 담아서 전달할 때 사용한다.
