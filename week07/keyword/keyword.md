- 미들웨어
    - **정의**: 요청(Request)과 응답(Response)의 중간 주기에서 특정 목적의 비즈니스 로직이나 검증을 수행하는 함수들의 체인(Chain) 구조 시스템입니다.
    - **속성 및 동작**: Express 기준 `(req, res, next) => void` 구조를 가지며, `next()` 함수를 호출하여 제어권을 다음 미들웨어 혹은 라우터 핸들러로 위임합니다.
    - **장단점**:
        - **장점**: 인증/인가, 로깅, CORS 설정 등 공통 관심사(Cross-cutting Concerns)를 분리하여 코드 중복을 방지하고 유지보수성을 극대화합니다.
        - **단점**: 미들웨어 체인이 복잡하고 길어질 경우, 디버깅 시 실행 흐름을 추적하기 어려워질 수 있습니다.
    - 라우터 핸들러마다 중복되는 유효성 검증이나 인증 로직을 한곳에서 일괄 처리할 수 있어 아키텍처를 단정하게 유지하는 데 핵심적인 요소라고 생각합니다.
- HTTP 상태 코드
    - **정의**: 클라이언트가 보낸 HTTP 요청이 서버에서 어떻게 처리되었는지 통보하는 3자리 숫자로 된 표준 응답 규격입니다.
    - **주요 대역별 속성**:
        - **2xx (Success)**: 요청 성공 (`200 OK`, `201 Created` 등)
        - **4xx (Client Error)**: 클라이언트의 요청 에러 (`400 Bad Request`, `401 Unauthorized`, `404 Not Found` 등)
        - **5xx (Server Error)**: 서버 내부 처리 에러 (`500 Internal Server Error`, `503 Service Unavailable` 등)
    - **장단점**:
        - **장점**: 프론트엔드와 백엔드 간 통신 규격을 표준화하여, 명세서 없이도 직관적으로 에러의 주체와 원인을 식별할 수 있습니다.
        - **단점**: 도메인별 세부 예외(비즈니스 예외)를 3자리 숫자만으로 모두 세밀하게 표현하기에는 한계가 있어, 통상 내부 커스텀 에러 코드를 바디에 병행 표기합니다.
    - 무조건 `200 OK`를 던지고 바디에 성공 여부를 적는 것보다, HTTP 스펙에 정의된 `400`이나 `404` 같은 상태 코드를 정확히 명시해 주는 것이 RESTful한 API 설계의 기본임을 체감했습니다.
- 에러 핸들링(Error Handling)
    - **정의**: 애플리케이션 실행 중 발생할 수 있는 예상치 못한 예외나 비즈니스 제약 조건 위배 상황을 포착하고 제어하여 시스템의 중단(Crash)을 막는 방어적 프로그래밍 기법입니다.
    - **속성**: Express에서는 인자가 4개인 에러 핸들링 전용 미들웨어 `(err, req, res, next) => void`를 라우터 최하단에 배치하여 센트럴라이즈드(중앙 집중형) 방식으로 예외를 통합 관리합니다.
    - **장단점**:
        - **장점**: 서버가 다운되는 것을 원천 방어하며, 사용자에게 시스템 내부 스택 트레이스 대신 정제된 에러 메시지를 안전하게 반환할 수 있습니다.
        - **단점**: 에러 핸들러 설계가 누락되거나 미들웨어 순서가 잘못 배치될 경우 에러가 유실되거나 화이트 스크린이 노출될 수 있습니다.
    - 데이터베이스 연동 과정에서 발생하는 `Timeout`이나 타입 충돌 같은 런타임 에러들을 클라이언트에게 노출하지 않고 글로벌 에러 핸들러에서 안전하게 가공하여 내려주는 파이프라인의 중요성을 배웠습니다.
- Tsoa
    - **정의**: TypeScript의 클래스와 데코레이터(Annotation) 설계를 기반으로 Express 라우터 제어와 OpenAPI(Swagger) 문서 생성을 자동화해 주는 프레임워크 라이브러리입니다.
    - **속성**: 컨트롤러 레이어 위에 `@Route`, `@Get`, `@Body` 등의 데코레이터를 작성하면 이를 컴파일하여 완전한 Swagger 스펙의 JSON/YAML 파일과 라우팅 코드를 생성(`tsoa spec-and-routes`)합니다.
    - **장단점**:
        - **장점**: 코드가 곧 명세서가 되므로 기획/코드 변경 시 Swagger 양식을 수동으로 업데이트해야 하는 번거로움과 싱크 불일치 문제를 완벽히 해결합니다. DTO 기반의 런타임 유효성 검증도 자동 처리됩니다.
        - **단점**: Tsoa 고유의 컴파일 프레임워크 규칙과 구조를 학습해야 하며, 프로젝트의 초기 보일러플레이트 설정이 다소 무겁고 복잡할 수 있습니다.
    - API 명세서 작성과 리팩토링은 백엔드 개발자에게 큰 비용인데, 소스 코드에 타입과 주석을 추가하는 행위 자체만으로 프론트엔드와 공유할 Swagger 페이지가 자동 갱신된다는 점에서 생산성을 획기적으로 올려주는 도구라고 생각합니다.
